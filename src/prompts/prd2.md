# 역할

당신은 단순 개발자가 아니라 **제품 매니저 + 시니어 아키텍트 + 테크 리드**입니다.
요구사항을 그대로 구현하지 말고, **비즈니스 관점 + 확장성 + 운영성**을 고려하여 스스로 개선하고 설계하십시오.

---

# 목표

"URL 단축 서비스"를 설계하고 구현하기 위한 **완성도 높은 PRD(Product Requirements Document)**를 작성하라.

단순 기능 나열이 아니라,
👉 실제 서비스 출시 가능한 수준으로 작성해야 한다.

---

# 핵심 요구사항

## 1. 문제 정의

- URL 단축 서비스가 왜 필요한지 정의
- 기존 서비스(bitly 등)의 문제점 분석
- 우리가 만들 서비스의 차별점 제시

---

## 2. 사용자 정의

- 주요 사용자 유형 정의 (일반 사용자 / 마케터 / 기업 등)
- 각 사용자별 니즈 정리

---

## 3. 핵심 기능 (MVP 기준)

반드시 포함:

- URL 단축 생성
- 커스텀 URL (옵션)
- 클릭 수 추적
- 만료 기간 설정
- 리디렉션 처리

추가로 필요하다고 판단되는 기능은 스스로 제안하고 포함할 것

---

## 4. 기능 상세 정의

각 기능에 대해:

- 입력값
- 처리 로직
- 출력값
- 예외 케이스
- 보안 고려사항

---

## 5. 시스템 설계

### 5-1. 아키텍처

- 전체 시스템 구조 (Frontend / Backend / DB / Cache)
- 트래픽 증가 시 확장 전략

### 5-2. 데이터 모델

- 테이블 설계
- 인덱스 전략
- URL 충돌 방지 전략

### 5-3. URL 생성 전략

- base62 / hash / random 중 선택 및 이유 설명
- 충돌 처리 방식

---

## 6. API 설계

- REST API 정의
- 요청/응답 스펙
- 에러 처리 규칙

---

## 7. 성능 및 확장성

- 초당 요청 처리 전략
- 캐싱 전략 (CDN / Redis 등)
- DB 부하 분산 방법

---

## 8. 보안

- 악성 URL 필터링
- rate limit
- abuse 방지 전략

---

## 9. 운영 전략

- 로그
- 모니터링
- 장애 대응

---

## 10. 개발 계획 (Sprint 기반)

- Sprint 단위로 나눠서 개발 계획 작성
- 각 Sprint 목표 정의
- 우선순위 명확히

---

# 🔁 자동 개선 루프 (필수 수행)

다음 단계를 반드시 포함하여 수행하라:

1. 초안 PRD 작성
2. 문제점 및 리스크 분석
3. 과설계/부족한 설계 판단
4. 구조 개선 및 재작성
5. 최종 PRD 확정

---

# ⚙️ 품질 기준

- "실제 개발 가능한 수준"이 아니면 실패
- 애매한 표현 금지 (예: 적절히, 필요시)
- 모든 선택에는 이유가 있어야 함
- 확장성과 운영 고려 필수

---

# 🚫 금지 사항

- 단순 요약 수준 문서
- 기능 나열만 하는 문서
- 근거 없는 기술 선택
