# 프롬프트 성능 비교 테스트 — 최종 결론

> 2개 PRD(URL 단축 서비스, 실시간 채팅앱) × 11개 프롬프트 × 총 14회 실행 기반

---

## 1. 핵심 발견

### "가장 좋은 프롬프트는 프롬프트를 쓰지 않는 것이다"

- **p2("잘 구현해줘")**가 코드 품질 평가에서 **14회 중 7회 1등** (평균 50/60)
- 역할 지정, 오케스트레이션 전략, 행동 지침 등 정교한 프롬프트들이 단 세 글자에 졌다
- Claude는 이미 "잘 만드는 법"을 알고 있으며, 지시하면 오히려 편향이 생긴다

### 프롬프트보다 PRD가 중요하다

- PRD에 "PORT 환경변수 필수"를 한 줄 추가하면 → 거의 모든 프롬프트가 지킴
- 프롬프트에 "보안 검증하라"를 써도 → 42/60, 여전히 취약점 존재
- **"뭘 만들지"(PRD)가 "어떻게 만들지"(프롬프트)보다 결과에 훨씬 큰 영향**

## 3. 핵심 인사이트

### 단어 하나가 결과를 바꾼다

| 프롬프트 | 비용 | 점수 |
|---------|-----|------|
| "**잘** 구현해줘" (p2) | $0.39 | **50** |
| "**간결하게** 구현해줘" (p9) | $0.22 | **32** |

- "잘" → "제대로 해라" → 만들고 확인하고 고치는 루프 발생
- "간결하게" → "적게 해라" → 핵심 기능 생략, 검증 안 함

### 역할 지정은 편향을 만든다

- "실용주의 개발자" → 속도 우선, 보안 무시
- "PM + 아키텍트" → 구조 과잉, CSS 깨짐
- "재귀 오케스트레이터" → 불필요한 분해
- **역할명을 빼면 품질이 올라간다** (p7 42점 → p8 48점)

### 검증 루프가 품질을 결정한다

| 검증 횟수 | 평균 점수 |
|----------|---------|
| 0회 (p6, p9) | 35 |
| 1회 (p8) | 48 |
| 3~4회 (p2) | 50 |

- 턴 수 자체가 중요한 게 아니라, **검증에 쓰인 턴**이 품질을 올린다
- 1턴 제한은 최악. 적절한 턴 + 검증이 최적

### 병렬 오케스트레이션은 이 규모에서 역효과

| | 비용 | 점수 |
|--|------|------|
| p1 (Agent 2~3개 위임) | $0.47 | 47 |
| p2 (혼자 다 함) | $0.39 | **50** |

- 분해 + 통합 비용 > 병렬 이득
- 캐시가 에이전트마다 별도 → 재활용 불가
- 컨텍스트 하나에 들어가는 작업은 쪼개지 않는 게 낫다

### 캐시 히트가 높다 ≠ 효율적이다

- 캐시 히트 = 이전 턴의 대화를 다시 읽는 것 (비용 발생)
- 턴이 많으면 캐시 히트 높지만 총 비용도 증가
- **적당한 턴 + 높은 캐시 히트율**이 최적

## 4. 실용적 가이드

### 프로토타입/PoC를 빠르게 만들 때

```
프롬프트: "잘 구현해줘" (p2)
PRD: 기능 목록 + 제약조건만 간결하게
```

### 비용을 아끼면서 괜찮은 품질을 원할 때

```
프롬프트: p8 (역할명 없는 행동 지침)
PRD: 기능 + 제약 + 보안 요구사항을 PRD에 명시
```

### 프로덕션급 코드가 필요할 때

```
프롬프트: p11 (범용 지침 + "사용자 입력 검증하라")
PRD: 기능 + 제약 + 보안 + 품질 기준을 구체적으로
```

### 어떤 경우든

- **PRD에 투자하라.** 프롬프트 다듬는 시간에 PRD를 다듬어라.
- **역할명을 쓰지 마라.** Claude의 기본 판단을 방해한다.
- **"어떻게"가 아니라 "뭘"을 알려줘라.** Claude는 이미 어떻게 만드는지 안다.

## 5. 한계 및 향후 과제

- **모델**: Claude Sonnet 4.6 기준. 다른 모델에서는 결과가 다를 수 있음
- **규모**: 단일 서비스 규모. 마이크로서비스 5개+ 규모에서는 오케스트레이션이 유효할 수 있음
- **코드 리뷰**: Claude가 Claude를 평가하는 구조. 객관성 한계 있음
- **샘플 수**: 프롬프트당 2~6회. 통계적 유의성에는 부족
