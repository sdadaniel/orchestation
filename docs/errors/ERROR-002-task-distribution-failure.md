# ERROR-002: 태스크 분배 전략 실패 (Task Distribution Failure)

## 개요
`TASK-001` 설계 시 오케스트레이션의 핵심인 '병렬성'과 '작은 단위의 작업(Atomicity)'을 고려하지 못하여, 시스템 전체 효율이 저하되고 비용이 폭발하는 결과 초래.

## 발생 일시
2026-03-31

## 상세 분석
1. **병렬 처리 기회 상실 (Loss of Parallelism)**:
   - `TASK-001`에 너무 많은 기능(DB, Auth, API 27개, Scripts)을 몰아넣음으로써, 하위 태스크들이 병렬로 실행될 기회를 차단함.
   - 오케스트레이터의 멀티 슬롯(Parallel Slots) 기능을 활용하지 못하고 거대한 단일 프로세스(Serialized Bottleneck)로 동작하게 만듦.

2. **지연된 피드백 루프 (Delayed Feedback Loop)**:
   - 태스크가 거대할수록 전체 작업이 100% 완료된 후에야 리뷰(Review)가 시작됨.
   - 중간 단계(예: DB 설계 단계)에서의 오류나 예산 초과를 즉시 감지하지 못하고, 모든 비용이 소모된 마지막 시점에야 실패를 확인하게 됨.

3. **인지 과부하 및 컨텍스트 비대화 (Context Bloat)**:
   - 단일 AI 작업자에게 수십 개의 파일 생성을 맡김으로써 컨텍스트가 임계치를 초과함.
   - 이로 인해 토큰 당 단가가 상승하고 AI의 정확도가 떨어지는 '비경제적 구간'에 진입함.

## 개선 가이드 (Task Design Principles)
- **10/10 원칙**: 하나의 태스크는 가급적 10개 내외의 파일 수정, 10분 내외의 실행 시간을 목표로 할 것.
- **PR 단위 쪼개기**: 실무에서 개발자가 하나의 Pull Request로 올릴 만한 적정 규모로 태스크를 분할할 것.
- **의존성 그래프 최적화**: 
  - 공통 기반(Base) 작업은 최대한 빠르게 끝내고, 
  - 그 이후 기능 단위(Feature-based) 작업들은 병렬로 나열하여 동시에 실행되도록 설계할 것.

## 결론
이번 사례는 **"오케스트레이터의 도구적 성능보다 설계자의 태스크 분배 역량이 프로젝트의 비용과 속도를 결정한다"**는 교훈을 남김.
