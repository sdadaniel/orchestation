# 오케스트레이션 성능 최적화 TODO

현재 `orchestrate.sh`의 배치 기반 실행 구조에서 발견된 병목과 개선 방안.

---

## 1. 이벤트 기반 스케줄링 (임팩트: 최대)

**현재 문제:** 배치 단위로 대기 — A가 1분에 끝나도 B가 10분이면 9분 낭비
```
[A,B,C 배치] → A끝(1분) → 대기... → B끝(5분) → C끝(5분) → [다음 배치]
```

**개선:** 태스크 하나 끝나면 즉시 다음 태스크 시작
```
[A,B,C 시작] → A끝(1분) → D 즉시 시작 → B끝(5분) → E 즉시 시작 → ...
```

- [ ] 배치 루프 → 이벤트 루프로 전환
- [ ] 태스크 완료 시 의존성 재평가하여 즉시 실행 가능한 태스크 큐잉
- [ ] 동시 실행 수 제한 (MAX_PARALLEL) 옵션

---

## 2. 머지 분리 (임팩트: 중)

**현재 문제:** 배치 내 모든 태스크가 끝나야 머지 시작 → 다음 배치 블로킹

**개선:** 태스크 완료 즉시 머지, 다음 태스크가 기다리지 않음

- [ ] 태스크 완료 콜백에서 바로 머지 처리
- [ ] 머지 충돌 시 자동 재시도 또는 큐잉
- [ ] 머지 순서 보장 (의존 관계 순)

---

## 3. 리뷰 병렬화 (임팩트: 중)

**현재 문제:** task → review → 수정 → review가 한 프로세스 안에서 직렬 실행

**개선:** 리뷰어를 별도 프로세스로 분리

- [ ] 태스크 완료 시 리뷰 프로세스 자동 스폰
- [ ] 리뷰 진행 중에도 다른 태스크 실행 가능
- [ ] 수정 요청 시 피드백 파일로 비동기 전달

---

## 4. 캐시 최적화 (임팩트: 높음, 난이도: 높음)

**현재 문제:** 매 태스크마다 프로젝트 컨텍스트를 새로 로드

**개선:** 같은 프로젝트 컨텍스트를 공유하는 태스크끼리 세션 재활용

- [ ] Claude 세션 캐시 전략 조사
- [ ] 프로젝트 컨텍스트 프리로딩
- [ ] 관련 태스크 그룹핑하여 같은 세션에서 실행

---

## 우선순위

1. **이벤트 기반 스케줄링** — 대기 시간 대폭 감소, 가장 체감 큰 개선
2. **머지 분리** — 비교적 간단, 파이프라인 처리량 증가
3. **리뷰 병렬화** — 리뷰 대기 제거
4. **캐시 최적화** — 비용/속도 개선, 조사 필요
