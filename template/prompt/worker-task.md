## 작업 규칙
- 이 Worktree 안에서만 코드를 수정한다
- main 브랜치를 직접 수정하지 않는다
- Task 상태를 완료 처리하지 않는다
- 작업이 끝나면 변경사항을 커밋해라
- orchestrate.sh, run-pipeline.sh, night-worker.sh를 절대 실행하지 않는다

## 작업 방식 — 즉시 실행, 최소 탐색
- **첫 턴에 바로 코드를 수정해라.** 분석/계획/설명에 턴을 쓰지 마라.
- scope에 있는 파일만 수정해라. scope 외 파일은 읽기만 허용한다.
- 같은 파일을 2번 이상 읽지 마라. 한 번 읽고 바로 수정해라.
- 프로젝트 구조를 탐색(find, ls)하는 데 2턴 이상 쓰지 마라. scope 파일을 바로 열어라.
- 확신이 없어도 먼저 수정하고 빌드/실행으로 검증해라.

## 시간 제한
이 작업은 최대 10분 안에 완료해야 한다. 10분 초과 시 강제 종료된다.
- 불필요한 파일 탐색을 하지 마라. scope 파일에 집중해라.
- 테스트 실행 → 실패 → 수정 루프를 3회 이상 반복하지 마라. 3회 안에 못 고치면 현재까지 작업을 커밋하고 종료해라.
- 작업이 10분 안에 끝나지 않을 것 같으면 가장 핵심적인 부분만 완료하고 나머지는 후속 태스크로 남겨라.

## 사전 확인 (필수 — 코드 수정 전에 반드시 실행)
작업을 시작하기 **전에**, scope 파일을 열어 이 태스크의 완료 조건이 **이미 충족되어 있는지** 확인해라.
이미 구현되어 있다면 코드를 수정하지 말고 거절해라 (아래 "거절 판단" 참조).

## 거절 판단
이 Task가 아래 조건에 해당하면 작업하지 말고 거절해라.

거절 조건:
- 사전 확인에서 이미 구현된 것으로 확인된 경우
- 요청 자체가 모순되거나 실현 불가능한 경우
- scope 파일이 존재하지 않거나 관련 없는 경우

해당하지 않으면 정상적으로 작업을 수행해라.

## 최종 출력 형식 (필수 — 반드시 지켜라)
작업이 끝나면 (성공이든 거절이든) result에 **JSON만** 출력해라.
설명, 요약, 체크리스트, 인사말 등 JSON 외의 텍스트를 절대 포함하지 마라.
파싱이 깨지면 작업 전체가 실패 처리된다.

성공 시 (이 형식 정확히):
{"status":"completed","summary":"작업 내용 한줄 요약","files_changed":["변경한 파일 경로"]}

거절 시 (이 형식 정확히):
{"status":"rejected","reason":"거절 사유"}
